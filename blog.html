Understanding Zero Trust Architecture in Cloud Identity Management

This post explains the main ideas behind Zero Trust security. It shows how companies can move away from risky passwords and switch to password-free login methods, like passkeys and biometric multi-factor authentication (MFA).

When I was selecting a focus area for my Final Year Project (FYP), I wanted to tackle a critical flaw in modern corporate networks: the reliance on weak, easily compromised password systems. With cloud infrastructure expanding rapidly, traditional perimeter security is no longer enough to keep malicious actors out.

Researching modern authorization structures led me straight to the core philosophy of Zero Trust: "Never Trust, Always Verify." Instead of assuming everything inside a network is safe, a Zero Trust architecture evaluates every single access request dynamically, protecting cloud infrastructure against advanced credential attacks.

To put these theories into practice, I engineered a fully functioning, passwordless multi-factor authentication and authorization framework. I scaled the architecture to simulate real-world enterprise constraints, executing threat modeling and vulnerability assessments to verify its resistance to interception. The deployment utilizes Python backend scripts to monitor anomalies, alongside cloud tunnels for secure data routing.

The Zero Trust Core Setup

Here is a quick breakdown of the primary tools and concepts I implemented to secure the authentication pipeline:

  • Keycloak (MFA & Passkeys): This server serves as the centralized identity provider. By removing standard passwords and relying on WebAuthn standards, users log in securely using hardware-bound passkeys and biometric validation (like fingerprints), neutralizing phishing and credential-stuffing risks.
  • Docker & AWS Infrastructure: The entire identity framework is containerized using Docker to ensure environment isolation and secure, repeatable deployments across Amazon Web Services (AWS) infrastructure environments.
  • Cloudflare Tunnel: To safely expose the authentication services without opening risky, public inbound firewall ports, I set up Cloudflare Tunnels. This establishes a secure, encrypted outbound connection directly to the edge network, minimizing the overall attack surface.

How to Secure Your Cloud Storage Using AWS IAM and Encryption

Learn how to set up AWS S3 bucket rules, create secure IAM user roles, and use AES-256 encryption to keep sensitive files safe from unauthorized access.

Cloud storage is incredibly convenient, but a single misconfiguration can expose sensitive enterprise data to the entire public internet. To explore how to solve this issue, I worked on a personal project to deploy an independent, hyper-secure file-sharing cloud platform.

The foundation relies on Amazon Web Services (AWS) S3 buckets combined with strict Identity and Access Management (IAM) policies. By default, public access to the storage must be completely blocked. Access permissions should follow the Principle of Least Privilege, ensuring that users or connected services only get the absolute minimum access required to complete their tasks.

Cloud Security with AWS

ResearchGate publication | DOI / full-text access

Implementing Server-Side Encryption and Governance

To protect data both while traveling across the network and while stored on disk, I set up the following layers of security:

  • Data Encryption (AES-256): All files stored within the cloud bucket are encrypted using server-side AES-256 encryption standards. Even if a bad actor manages to physically access the underlying storage disks, the data remains unreadable text without the proper decryption keys.
  • Link Security Controls: When files are shared externally, the application enforces automated link expiry timelines, custom password protections, and strict download restrictions to maintain continuous data ownership.

Building this environment highlights how proper data governance configurations can successfully stop unauthorized cloud data exfiltration.

Network Logging and Traffic Analysis for Beginners Using Wireshark

Discover how to capture live data packets, analyze network logs with Wireshark, and use tools like Nmap to spot suspicious activity and fix security weak points.

Every activity on a computer network leaves a digital footprint. As an aspiring Cloud SOC Analyst, learning how to read these footprints through packet analysis is one of the most critical skills you can develop to trace network intrusions.

Using Wireshark, an industry-standard network protocol analyzer, you can intercept and view live data traffic running across a network line. This lets you look deep into network packets to see exactly what applications are doing, track communication protocols (like TCP/IP, DNS, or HTTP), and isolate malicious anomalies.

Network Traffic Analysis Using Wireshark

Key Steps for Basic Network Monitoring

When analyzing a network for security vulnerabilities, a standard monitoring workflow usually follows these phases:

  • Network Reconnaissance (Nmap): Before looking at individual packets, use Nmap to scan your network environment. This helps you map active devices, discover open communication ports, and detect unauthorized active systems.
  • Traffic Filtering & Triage: A live network generates thousands of log entries every second. Learning to apply specific display filters in Wireshark—such as tracking suspicious IP addresses or detecting unusual TCP handshake flags—helps security analysts quickly cut through false alarms to spot actual threats.

Mastering these foundational packet-hunting techniques is a massive advantage when preparing for practical industry challenges or cybersecurity hackathons.
